## Top-level structure

All simulation data lives in `data\`, deliberately separated from `code_insights\`
to prevent data files from being indexed into Claude's context.
**All new sweep output must also be written here** — never write data inside `code_insights\`.


## Authority rules (the part Claude must respect)

1. **base.toml is the single source of truth** for platform constants and solver
   settings. PlatformParams and all TOML generators read it with NO fallbacks.
   Editing a physics constant ⇒ regenerate the gated profile TOMLs
   (gen_ellipse_toml.py, gen_spiral_orbit_toml.py, gen_multisine_toml.py).
2. **Module copies beside the notebook are authoritative.** If a chat produces an
   updated profiles.jl/datastore.jl, it replaces the copy at the project root AND
   gets re-uploaded to project knowledge in the same session. Any other copy is
   stale by definition.
3. **Everything runs from the project root** (notebook kernel pwd = driver pwd =
   code_insights\). CONFIG_DIR = "trajectory_files" resolves identically for the
   notebook and the sweep driver — one relative path, no translation needed.
4. Output files are never hand-edited; resume logic treats an existing .arrow
   as a completed job (atomic tmp-then-rename guarantees no partials).
5. Only the notebook's **Cell 2** carries the `parameters` tag — multiple tagged
   cells break extract_run_one.py.

## Known landmines
- OneDrive sync can touch mtimes mid-session (precompile cache noise).
- Two stale-copy incidents have already occurred (profiles.jl, datastore.jl) —
  rule 2 exists because of them.
- Multiple cells tagged `parameters` breaks extract_run_one.py: ONLY Cell 2.
